  • 아래와 같은 상속 구조를 갖는 상황에서 아래 코드들은 컴파일이 가능할까요?

String s = "35";
Integer x = (Integer s);
x.floatValue();

Number x = new Double(3.5);
Integer z = (Integer) x;

결론부터 말하자면, String → Integer Casting은 에러, Double → Number Casting은 가능합니다.

상속 구조를 다시 한 번 살펴보면서 이유를 살펴봅시다

Integer와 Number는 직접적인 상속 관계에 있는 반면, String과 Integer는 Object에서 파생되었다는 것 이외에는 직접적인 상속 관계를 갖지 않습니다. 따라서, String에서 Integer로 직접적인 캐스팅은 불가, 컴파일 에러가 발생합니다.

반면, Number에서 Integer로의 캐스팅은 컴파일 에러가 발생하지 않습니다. (앞서 이야기한 것처럼, 직접적인 상속 관계에 있기 때문입니다.)

만약, String과 Integer 사이의 변환을 하고 싶다면, 둘의 공통 상위 클래스가 되는 Object를 통해 캐스팅해야 합니다. (일전 이야기한 것과 같이 런타임 에러가 발생하지 않는다고 해서 코드가 안전하리라는 보장은 없습니다. 런타임 에러는 얼마든지 발생할 수 있습니다.)

String s = "35";
Object o = s;
Integer i = (Integer)o;

예제를 하나 더 살펴봅시다.

아래 코드는 컴파일이 가능할까요?

Number x = new Double(3.5);
Integer i = (Integer) x;

Double → Number, Number → Integer 모두 직접적인 상속 관계를 갖고 있지요? 따라서 컴파일 에러는 발생하지 않습니다.

다소 복잡한 내용이었으며, 그림을 통해 정리해보면 다음과 같습니다.

아래에서 윗 가지의 Class로 변환하는 것은 is-a 관계가 성립하기 때문에 캐스팅이 필요 없습니다.

반면, 위에서 아래 가지로의 변환은 캐스팅이 필요하며, 컴파일 에러는 막을 수 있지만 런타임 에러까지 보장하지는 않습니다.

직접적으로 같은 라인에 연결되어 있지 않은 String, Integer 사이의 변환 같은 경우, 캐스팅을 사용해도 변환이 불가합니다. (공통분모인 Object를 거쳐야 가능하지요.)

Hiding

static method나 변수를 overriding할 수 있을까요? 이를 별도로 hiding이라고 부르며, 사실은 비추천되는 구현 방식입니다.

// Animal.java
public class Animal {
    public static void bark(){
        System.out.println("Dog bark");
    }
}

// Dog.java
public class Dog extends Animal {
    @Override
    public static void bark(){
        System.out.println("Animal Bark");
    }
}

왜 hiding이 비추천되는지 차근차근 생각해 봅시다.

  • static method를 사용하는 이유는 본질적으로 클래스 instance 없이도 해당 method를 사용하기 위함입니다.
  • 상속을 통해 override를 하는 이유는 instance의 dynamic type에 맞는, dynamic method를 사용하기 위함이지요.

이렇게 static method와 상속은 정반대의 목적을 위해 사용됩니다.

⇒ 따라서, 이 둘을 혼용하는 static method의 overriding, hiding은 비추천됩니다.

Subtype Polymorphism

Higher Order Function (HOF라고 줄여 말하겠습니다.)을 사용하여 tenX를 구현했던 내용을 기억하시나요?

import java.util.function.Function;

public class HoFDemo {

    public static int tenX(int a){
        return a * 10;
    }

    public static int doTwice(Function<Integer, Integer> f, int a){
        return f.apply(f.apply(a));
    }

    public static void main(String[] args) {
        int result = doTwice(HoFDemo::tenX, 2);
        System.out.println("Result is " + result);
    }
}

이번에는 다형성(Polymorphism)을 사용해서 다양한 로직을 구현해보고, Higher Order Function과 장단점을 비교해 보겠습니다.

  • 아래의 파이썬 코드를 함께 살펴봅시다.
def print_larger(x, y, compare, stringify):
    if compare(x, y):
        return stringify(x)
    return stringify(x)

def print_larger(x, y):
    if x.largerThan(y):
        return x.str()
    return y.str()
  • HOF를 사용할 경우, 비교 대상인 x, y와 함수 compare, stringify를 모두 매개변수로 전달해야 하는 반면, Subtype Polymorphism을 사용할 경우, x, y 내부에 메소드를 구현한 뒤, 이 두 매개변수만 전달하면 같은 동작의 실행이 가능합니다.
  • 더불어, x, y 인스턴스를 사용하면서 상속의 개념을 더한다면, Integer, Float, Double과 같이 다양한 타입을 위해 overloading을 사용하지 않더라도 하나의 함수 구현으로 여러 상황을 대처할 수 있을 것입니다.

이번에는 Subtype Polymorphism을 사용하여, 어떠한 데이터 타입이 들어와도 동작하는 max() 메소드를 구현해보려 합니다. Primitive type은 물론이고, 다양한 Reference type들에 대해서도 대소관계를 정의하고 가장 큰 값을 찾아보려 합니다.

아래와 같이 max 코드를 구현해 보았는데, 코드에서 컴파일 에러가 발생할 여지는 없을까요? 생각해보면서 코드를 읽어봅시다.

// Maximizer.java
public class Maximizer {
	public static Object max(Object[] items){
		int maxDex = 0;
		for(int i = 0; i < items.length; i++)
			if(items[i] > items[maxDex])
				maxDex = i;
		return items[maxDex];
	}
}

// DogLauncher.java
public class DogLauncher {
    public static void main(String[] args) {
        Dog d1 = new Dog("Elyse", 3);
        Dog d2 = new Dog("Sture", 9);
        Dog d3 = new Dog("Benjamin", 15);
        Dog[] dogs = new Dog[]{d1, d2, d3};

        Dog d = (Dog) Maximizer.max(dogs);
        d.bark();
    }
}
  • Dog is-a Object이기 때문에 max()의 input 매개변수 타입이 Object[] 인 것은 괜찮습니다.
public class Maximizer {
	public static Object max(Object[] items){
  • max()의 return type은 Object이지만, Dynamic Type은 Dog이기 때문에, Dog으로 캐스팅하는 부분 또한 괜찮습니다.
Dog d = (Dog) Maximizer.max(dogs);

하지만, max 함수 내 Object끼리의 비교에서 > 연산을 사용할 때, 에러가 발생합니다. > 연산으로는 Object끼리의 비교할 수 없으며, 다른 말로 Java에서는 연산자 overloading을 허용하지 않습니다.

for(int i = 0; i < items.length; i++)
	if(items[i] > items[maxDex])
		maxDex = i;

이를 해결하기 위해서, maxDog이라는 메소드를 추가로 만들어보고자 합니다.

public static Dog maxDog(Dog[] dogs){
	if(dogs == null || dogs.length == 0 )
		return null;
	Dog output = dogs[0];
	for(int i = 0; i < dogs.length; i++)
		if(dogs[i].size > output.size)
			output = dogs[i];
	return output;
}

그런데, 여기서도 문제가 발생합니다. Cat, Turtle, Horse… 등 비교하고자 하는 동물들이 추가될 때마다 maxCat, maxTurtle, maxHorse와 같이 매번 추가 메소드를 모두 구현해줘야 한다는 귀찮은 문제가 발생합니다.

인터페이스와 상속을 통해 이 문제를 해결해봅시다!

Comparable

Java에는 compareTo()라는 메소드를 가진 Comparable 인터페이스가 있습니다.

public int compareTo(T o);

  • 이 인터페이스를 상속받으면, 클래스 간 비교를 하는 compareTo 메소드를 무조건 구현해야 합니다.

  • 그리고 두 Object간의 대소 관계가 결정되었기 때문에, Comparable인터페이스를 상속받은 모든 클래스들은 서로간의 비교가 가능합니다.

Comparable 인터페이스를 상속받도록 Dog Class를 수정해봅시다.

public class Dog implements Comparable<Dog> {
    ...
    @Override
    public int compareTo(Dog uddaDog) {
        //assume nobody is messing up and giving us
        //something that isn't a dog.
        return size - uddaDog.size;
    }
}

max() 메소드도 compareTo를 사용하도록 수정해봅시다.

public class Maximizer {
	public static Comparable max(Comparable[] items) {
		int maxDex = 0;
		for (int i = 0; i < items.length; i += 1)
			if (items[i].compareTo(items[maxDex] > 0)
				maxDex = i;

		return items[maxDex];
	}
}

만약, Cat, Turtle 등 동물들이 추가된다 하더라도 max() 메소드는 재사용이 가능합니다.

  1. Comparable 인터페이스를 상속받고,
  2. compareTo 메소드를 구현하면
  3. Comparable 타입을 받는 max를 사용 가능합니다.
public class Cat implements Comparable<Cat> {
	  @Override
    public int compareTo(Cat cat)
}

public class Turtle implements Comparable<Turtle> {
	  @Override
    public int compareTo(Turtle turtle)
}
...

이렇게 Subtype Polymorphism을 사용할 경우, 구현해야 하는 코드의 양도 줄어들고, 범용성도 높아진다는 것을 알 수 있습니다. 👍
