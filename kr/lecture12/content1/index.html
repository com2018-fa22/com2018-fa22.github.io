Type Checking and Casting

위 내용을 학습하기 위해서, DMS(Dynamic Method Selection), Static Type과 Dynamic Type, 그리고 상속의 is-a 관계에 대한 이해가 필요합니다.

퍼즐과 함께 Lecture12를 시작해봅시다. 기억할지 모르겠지만, 지난 시간 구현하였던 SLList, VengefulSLList 클래스를 사용하는 퀴즈입니다.

(VengefulSLList는 SLList를 상속받아 removeLast() 메소드를 overriding 하며, printLostItems라는 메소드를 추가로 갖고 있습니다.)

  • 아래 코드에서 에러가 발생할까요? 그렇다면 어디에서, 왜 에러가 발생할까요?

s1의 static type은 SLList이며, dynamic type은 VengefulSLList입니다. 따라서, s1.addLast()와 s1.removeLast()는 전혀 문제가 되지 않습니다.

위 코드에서 에러가 발생하는 부분은 크게 2가지 입니다.

s1.printLostItems()

s1의 static type은 SLList 입니다. 하지만, printLostItems은 VengefulSLList에만 존재하는 메소드이지요. 따라서 컴파일 에러가 발생합니다.

VengefulSLList<Integer> vsl2 = s1;

VengefulSLList is-a SLList는 성립하지만, SLList is-a VengefulSLList의 관계는 성립하지 않습니다!

하위 클래스에 상위 클래스 인스턴스를 대입할 수는 없지요. 상속 관계에서 컴파일 에러가 발생합니다.

마지막 라인의 에러를 해결해봅시다.

우리는 s1이 Dynamic Type으로 VengefulSLList을 갖는다는 것을 알고 있고, 또 이러한 구현이 필요합니다. 이때, casting을 사용하면, 마지막 줄의 컴파일 에러를 해결할 수 있습니다.

VengefulSLList<Integer> vsl2 = (VengefulSLList)s1;

여기에서 명심할 것은, 컴파일 에러는 발생하지 않지만 런타임 에러로부터 안전하리라는 것은 보장하지 못한다는 점입니다.

더불어, removeLast의 경우 VengefulSLList에서 override된 메소드는 VengefulSLList의 것을 사용하게 되고, 그렇지 않은 메소드는 SLList의 것을 사용하게 되는 점에서 OOP를 사용하는 것의 이점을 확인할 수 있습니다.

overriding과 상속은 코드의 복잡도를 줄여주는 키포인트가 됩니다.

How to Check Dynamic Type of Instance

static type과 dynamic type 때문에 발생하는 컴파일 에러는 모두 살펴보았습니다.

그러면, 런타임 에러는 어떻게 해결할 수 있을까요? 그 예방법으로 특정 인스턴스의 dynamic type을 확인하고 싶다면 다음과 같은 두가지 방법을 사용할 수 있습니다.

  1. instanceof 키워드 사용하기

instanceof 키워드는 특정 인스턴스의 dynamic type을 비교하여 true, false를 반환하는 키워드입니다.

public static void main(String[] args) {
    VengefulSLList<Integer> vs1 = new VengefulSLList<Integer>(0);
    SLList<Integer> s1 = vs1;

    if(s1 instanceof VengefulSLList)
        System.out.println("s1 instanceof VengefulSLList");
    ...
}

실행 결과
s1 instanceof VengefulSLList
  1. getClass() 메소드와 static 변수 class 사용하기

모든 클래스는 class라는 class static 변수를 갖고 있으며, 모든 인스턴스는 이 값을 반환하는 getClass()라는 메소드를 갖고 있습니다. 이를 통해 dynamic type을 얻을 수 있지요.

VengefulSLList()를 통해 생성된 모든 인스턴스들은 getClass() 실행 시 모두 같은 값을 반환할 것입니다. 이러한 상황을 singleton이라고 이야기한답니다.

public static void main(String[] args) {
    VengefulSLList<Integer> vs1 = new VengefulSLList<Integer>(0);
    SLList<Integer> s1 = vs1;

    System.out.println(s1.getClass());
    System.out.println(VengefulSLList.class);
    ...
}

실행 결과
class VengefulSLList
class VengefulSLList

Casting and Runtime Error, Precision

캐스팅 시 컴파일 에러가 발생하지 않는다고 해서 안전한 코드라고 할 수는 없습니다. 컴파일에는 성공하더라도 런타임 시 의도치 않은 결과를 얻을 수 있는 상황을 생각해봅시다.

// Animal.java
public class Animal {

    public int age;
    public Animal(){
        age = 0;
    }
}

// Dog.java
public class Dog extends Animal {

    public float tailLength;

    public Dog(){
        tailLength = 0.0f;
    }

    public float getTailLength(){
        return this.tailLength;
    }

    public static void main(String[] args) {
        Animal ani = new Animal();
        System.out.println(((Dog)ani).getTailLength());
    }
}

getTailLength()는 Dog class에만 존재하는 메소드입니다.

Animal 타입인 ani에서 캐스팅을 통해 getTailLength를 호출할 수는 있지만, 이때 얻게 되는 값은 의도치 않은 쓰레기 값을 얻게 될 것입니다. getTailLength는 Dog에만 존재하는 클래스 변수, tailLength값을 반환하기 때문이지요.

추가적으로, primitive type의 casting도 살펴봅시다.

Java는 매우 보수적이기 때문에 double ⇒ int, float ⇒ int와 같은 변환을 허용하지 않습니다.

하지만, 종종 프로그래머의 입장에서 정확도의 손실을 감수하더라도 이러한 변환을 사용하고 싶은 경우가 있습니다. casting을 통해 명시적으로 이러한 변환을 구현할 수 있습니다.

double d = 3.1415;
int i = (int)d;

위 코드의 실행 시, d의 소수점 아래 부분은 잘려나가고 i에는 3이 대입될 것입니다.

중간 정리

  • 변수의 instantiation, 대입 시 is-a 관계에 주의해야 합니다.
  • 상속 관계를 갖는 클래스들 사이의 변환 시, 명시적인 casting을 통해 컴파일 에러를 방지할 수 있지만 의도치 않은 결과에 주의해야 합니다.

Higher Order Functions (A First Look)

Higher Order Function이란, function을 하나의 변수처럼 다루는 것을 이야기합니다. C/C++에서는 함수의 포인터를 사용가능하기 때문에 functor라는 이름으로 주로 사용되며, 파이썬에서도 이러한 코드를 작성할 수 있지요.

  • Python Code
def tenX(x):
    return 10 * x

def do_twice(f, x):
    return f(f(x))

print(do_twice(tenX, 2)) # 200

그럼, Java에서는 이러한 구현이 가능할까요?

Java의 Memory Box는 오로지 변수를 담을 수 있다는 근본적인 제약 때문에 Java 7이전에는 interface를 사용하여 Higher Order Function을 구현하였습니다.

public interface IntUnaryFunction {
    int apply(int x);
}
...
public class TenX implements IntUnaryFunction {
    /** Returns ten times the argument. */
    public int apply(int x) {
        return 10 * x;
    }
}
...
public class HoFDemo {
    public static int do_twice(IntUnaryFunction f, int x) {
        return f.apply(f.apply(x));
    }

    public static void main(String[] args) {
        IntUnaryFunction tenX = new TenX();
        System.out.println(do_twice(tenX, 2));
    }
}
  • 인터페이스를 통해 메소드를 정의하고
  • 해당 인터페이스를 상속받아 메소드를 구현하는 클래스를 만들고,
  • 비로소 작성한 클래스를 통해 Higher Order Function를 사용 가능합니다.

매우 비효율 적이고 복잡하지요.

Java 8 이후부터, 앞선 복잡한 인터페이스 구현 없이도 Higher Order Function을 사용할 수 있게 되었으며, Function interface를 사용하여 이를 구현할 수 있습니다.

import java.util.function.Function;

public class HoFDemo {

    public static int tenX(int a){
        return a * 10;
    }

    public static int doTwice(Function<Integer, Integer> f, int a){
        return f.apply(f.apply(a));
    }

    public static void main(String[] args) {
        int result = doTwice(HoFDemo::tenX, 2);
        System.out.println("Result is " + result);
    }
}

실행 결과
Result is 200

Function<Integer, Integer>는 매개변수가 Integer이고, output이 Integer인 모든 Function을 지칭할 수 있는 타입이 됩니다.

만약 매개변수가 2개일 경우에는, Function이 아닌 BiFunction을 사용하며, BiFunction<param, param, output>의 형태로 사용할 수 있습니다.

import java.util.function.BiFunction;
import java.util.function.Function;

public class HoFDemo {

    public static int tenX(int a){
        return a * 10;
    }

    public static int mul(int a, int b){
        return a * b;
    }

    public static int doTwice(Function<Integer, Integer> f, int a){
        return f.apply(f.apply(a));
    }

    public static int doMultiply(BiFunction<Integer, Integer, Integer> f, int a, int b){
        return f.apply(a, b);
    }

    public static void main(String[] args) {
        int result = doMultiply(HoFDemo::mul, 7, 8);
        System.out.println("Result is " + result);
    }
}

매개변수가 3개 이상일 경우에는 자체적인 Interface를 구현하여 사용해야 합니다.
